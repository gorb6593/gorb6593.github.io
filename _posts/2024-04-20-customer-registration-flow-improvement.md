서비스 운영에서 눈에 띄는 문제는 기능 부족보다도 **등록 절차가 길고 사람이 반복해서 확인해야 하는 지점이 많다**는 점이었습니다.\
이 글은 실제로 고객 등록 시간을 줄이기 위해 어떤 문제를 정의했고, 어떤 방식으로 개선했는지를 정리한 예시입니다.

## 문제 상황

기존 등록 프로세스에는 다음 문제가 섞여 있었습니다.

- 수임동의 이후 다음 상태로 넘어가는 기준이 명확하지 않았습니다.
- 운영 담당자가 여러 화면을 오가며 수동으로 확인해야 했습니다.
- 누락이 생기면 고객 응답이 지연되고, 전체 처리 시간이 길어졌습니다.

결과적으로 등록 완료까지 걸리는 시간이 길었고, 시즌성 유입이 많을 때 병목이 더 크게 드러났습니다.

## 접근 방식

개선은 크게 세 단계로 나눴습니다.

### 1. 상태 전이를 먼저 정리

사람이 판단하던 흐름을 시스템이 처리할 수 있도록 등록 과정을 상태 중심으로 다시 정의했습니다.

- 어떤 이벤트가 발생하면 다음 상태로 이동하는지 명시했습니다.
- 예외 케이스를 먼저 분리해 운영자가 꼭 봐야 하는 건만 남겼습니다.
- 운영 화면에서도 현재 상태와 다음 액션이 바로 보이도록 맞췄습니다.

### 2. 반복 확인 작업 자동화

운영팀이 매번 확인하던 작업 중 규칙으로 처리 가능한 부분은 자동화했습니다.

- 특정 조건을 만족하면 고객 상태를 자동 변경
- 후속 처리 대상만 별도로 추출
- 누락 가능성이 높은 케이스는 알림으로 전환

### 3. 운영 기준을 문서화

프로세스 개선은 코드만으로 끝나지 않기 때문에, 운영 기준과 예외 처리 기준도 함께 정리했습니다.

- 신규 담당자 온보딩 시간을 줄일 수 있도록 절차 문서화
- 자주 발생하는 예외 상황에 대한 대응 기준 정리
- 기능 변경 시 함께 봐야 할 체크리스트 추가

## 결과

개선 이후 다음과 같은 변화를 만들 수 있었습니다.

- **고객 등록 시간 70% 단축**
- 반복 수작업 감소로 운영팀 처리 부담 완화
- 상태 누락과 커뮤니케이션 비용 감소

## 이 글을 템플릿으로 쓰는 방법

앞으로 홈에 연결할 상세 글은 아래 흐름으로 쓰면 됩니다.

1. 문제 상황
2. 원인 분석
3. 해결 방식
4. 결과 수치
5. 회고 또는 다음 개선 포인트

이 구조로 쓰면 채용 담당자나 동료가 봤을 때도, 단순 나열이 아니라 **무엇을 왜 개선했는지**가 바로 전달됩니다.
