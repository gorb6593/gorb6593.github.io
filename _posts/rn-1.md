이 글은 기웅정보통신에서 진행한 **요양 시스템 전면 리뉴얼** 작업을 돌아보며, 왜 이 프로젝트가 필요했고 어떤 기준으로 리뉴얼을 진행했는지 정리한 회고입니다.

## 배경

기존 시스템은 오랜 기간 운영되며 기능은 많이 쌓였지만, 유지보수와 확장에 부담이 커지고 있었습니다.

- 레거시 구조에서 기능 추가 속도가 점점 느려졌습니다.
- 화면과 서버 구조 모두 변경 비용이 커졌습니다.
- 운영 관점에서도 관리 포인트가 복잡해지고 있었습니다.

## 목표

이번 리뉴얼의 핵심은 단순 기술 교체가 아니라, **앞으로 계속 개발 가능한 구조를 만드는 것**이었습니다.

- PHP 기반 구조를 Java Spring Boot, Vue.js 기반으로 전환
- 관리자 기능과 외부 연동을 확장 가능한 형태로 정리
- 운영팀과 개발팀 모두가 다루기 쉬운 구조 확보

## 내가 맡은 역할

- 백엔드 구조 전환 주도
- 관리자 페이지 및 외부 연동 기능 개발
- 리뉴얼 과정에서 필요한 도메인 이해와 요구사항 정리

## 어떻게 진행했는가

### 1. 기존 구조 파악

리뉴얼에서 가장 먼저 한 일은 기능을 새로 만드는 것이 아니라, 기존 시스템이 어떤 흐름으로 동작하는지 정리하는 것이었습니다.

- 실제 운영 화면 기준으로 업무 흐름 파악
- 누가 어떤 기능을 쓰는지 확인
- 먼저 옮겨야 하는 핵심 기능 우선순위 정리

### 2. 기술 전환

기술 전환은 “최신 기술 사용”보다도 이후 유지보수 효율을 높이는 데 초점을 맞췄습니다.

- Spring Boot 기반 백엔드 구조 설계
- Vue.js 기반 화면 구성
- 기능 단위로 이관 범위를 나누고 점진적으로 정리

### 3. 연동과 운영 기능 보강

실무 시스템은 핵심 도메인 기능만으로 끝나지 않기 때문에 주변 운영 기능도 함께 중요했습니다.

- OzReport 연동
- 내부 관리자 페이지 개발
- 공단 스크래핑 연동

## 결과

- 레거시 구조에서 벗어나 이후 기능 확장이 가능한 기반 마련
- 관리자 기능과 외부 연동을 더 일관된 구조로 정리
- 운영 시스템 특성상 필요한 업무 흐름을 개발 구조에 반영

## 아쉬웠던 점

- 레거시 시스템 특성상 숨어 있는 예외 케이스가 많아 예상보다 도메인 파악 비용이 컸습니다.
- 리뉴얼 초기에 더 많은 운영 데이터와 사례를 함께 분석했으면 전환 속도를 더 높일 수 있었을 것 같습니다.

## 회고

전면 리뉴얼은 기술 교체보다 **도메인 재이해와 구조 재정의**가 더 큰 비중을 차지하는 작업이었습니다.  
이번 경험을 통해, 운영 서비스 리뉴얼은 결국 “무엇을 새로 만들 것인가”보다 “무엇을 유지하고 무엇을 버릴 것인가”를 명확히 정하는 일이 중요하다는 점을 다시 확인했습니다.
