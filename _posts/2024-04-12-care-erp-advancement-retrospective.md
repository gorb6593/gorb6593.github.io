이 글은 리뉴얼 이후 진행한 **요양 시스템 고도화** 작업을 중심으로, 기능을 더 붙이는 수준이 아니라 실제 운영 가치를 높이기 위해 무엇을 고민했는지 정리한 회고입니다.

## 배경

전면 리뉴얼 이후 시스템의 기반은 갖춰졌지만, 실제 운영 현장에서 필요한 추가 기능은 계속 생기고 있었습니다.

- 결제 연동 필요
- 관리자 기능 확장 요구
- AI 기반 업무 지원 기능 검토
- 세무 서비스 연계 필요

## 목표

- NICE 결제 연동
- 관리자 기능 보강
- AI 업무 지원 기능 추가
- 세무 서비스 확장 기반 마련

## 내가 맡은 역할

- 고도화 범위의 기술 검토
- 기능별 우선순위 정리와 구현
- 기존 리뉴얼 구조 위에서 추가 기능이 무리 없이 붙도록 설계

## 어떻게 진행했는가

### 1. 운영 관점에서 기능 우선순위 정리

고도화 작업은 요청이 많다고 다 먼저 하는 것이 아니라, 운영 임팩트가 큰 순서대로 정리하는 것이 중요했습니다.

- 결제 기능처럼 바로 운영 프로세스에 영향을 주는 기능 우선
- 관리자 기능은 실제 사용하는 흐름 기준으로 보완
- AI 기능은 실무 효율이 높은 보조 영역부터 접근

### 2. 기존 구조를 해치지 않는 확장

리뉴얼 직후 고도화가 붙으면 구조가 다시 어지러워지기 쉽기 때문에, 기존 구조와의 정합성을 계속 확인했습니다.

- 결제 연동 흐름 명확화
- 관리자 기능을 도메인 흐름 안에 맞게 배치
- 새로운 기능이 기존 운영 로직을 깨지 않도록 검증

### 3. 서비스 확장 관점 고려

단기 요청을 처리하는 동시에, 이후 다른 서비스가 붙을 가능성도 함께 고려했습니다.

- 세무 서비스 확장 가능성 검토
- 데이터 연결 지점과 관리 포인트 정리

## 결과

- 운영 가치가 큰 기능을 순차적으로 보강
- 리뉴얼 이후 구조를 유지하면서 기능 확장 가능
- 단발성 기능 추가가 아니라 다음 단계 확장을 고려한 기반 정리

## 아쉬웠던 점

- 고도화 단계에서는 요구사항이 빠르게 바뀌는 경우가 많아, 초기 명세가 자주 흔들렸습니다.
- 이후에는 기능 도입 전에 운영 지표와 기대 효과를 더 명확히 합의하면 좋겠다고 느꼈습니다.

## 회고

고도화 작업은 단순 기능 추가가 아니라 **“지금 필요한 것”과 “나중에도 유지될 것” 사이의 균형을 잡는 일**이었습니다.  
이번 경험을 통해, 운영 시스템의 확장은 기술보다도 우선순위 판단과 구조적 일관성이 더 중요하다는 점을 다시 확인했습니다.
